# i18next-kit v0.2 / Plugin 重构说明

本文档合并 **v2 产品决策**、**实现风险**与**推荐重构顺序**，供开发与验收对齐。

---

## 1. 原始目标（持续更新）

1. **chokidar**：引入并专职监听 i18n 契约与各 locale（**待实现**：插件已重置为仅 `index.ts` 空壳，从此处重新接入）。
2. **插件入口**：Vite 侧 `i18nextKit(options)` 的 **options 保持兼容**；新能力通过**可选字段**扩展。
3. **dev 启动与脚手架**：`pnpm dev` 后若不存在 i18n / locale 布局，则创建最小模板；**默认**落在仓库根下 **`src/...`**；用户可通过配置指定根下**任意**相对路径。
4. **base ↔ 各 locale 同步**：监测 `base`（契约目录）的增删改（及重命名），**自动**在各 `<locale>` 目录做对应文件的生成、覆盖、删除、改名；**不写死**如 `hello` 等示例文案，按推断生成（见 §3）。
5. **MVP 体验**：暂时**不要**终端错误样式提示与 **overlay**，快速迭代；边界与严格模式后续再补。

---

## 2. 已拍板决策一览

| 主题 | 结论 |
|------|------|
| **framework** | 增加 **`framework`** 配置；**默认**在根目录下使用 **`src/...`** 相关默认路径；用户通过 **`i18nDir`**（及现有 `contractsDir` / `outDir` 等）指定根下**任意**相对路径；**生成与 base↔locale 同步**一律基于 **`resolveConfig` 解析后的绝对路径**。 |
| **base ↔ locale：新增** | `base/foo.ts` 新增时，**总是**为每个配置的 locale 生成 `foo.ts` 模板；**不写死** `hello`；按**推断**生成；若契约侧刚开始等价于**空对象**，则各 locale 亦为**空 default 对象**。 |
| **base ↔ locale：修改** | `base/foo.ts` 修改时，**自动**修改各 locale 的 `foo.ts`；若用户已在 locale 中手写文案，对 **default 对象一律覆盖**（以 base 推断结果为准）。 |
| **base ↔ locale：删除** | 删除 `base/foo.ts` 时，**自动删除**各 locale 下的 `foo.ts`。 |
| **base ↔ locale：重命名** | `base` 下 `foo.ts` → `foo1.ts` 时，各 locale 下同步 **`foo.ts` → `foo1.ts`**。 |
| **代码风格** | locale 侧示例与生成物统一采用 **`satisfies XxxMessage`**，收紧类型。 |
| **终端 / overlay** | MVP 阶段：**不要** overlay 与「终端错误式」提示；后续再区分 dev 提示与 `build` 严格失败等。 |
| **与 generateAll** | **base 驱动 locale 文件**与 **`generateAll`（contracts、generated-resources 等）** 同属一条流水线：**先**完成磁盘上的 locale 同步（或 noop），**再**调用 **`generateAll`** 生成产物。 |

---

## 3. base ↔ locale 行为细则（验收口径）

- **命名与类型名**：与现有约定一致，例如 `user-management.ts` → `UserManagementMessage`（见 `scan-contracts`）。
- **新增 namespace 文件**：为每个 `locales[]` 写入 `import type { … } from "../base/<ns>"`（路径深度按实际相对 `base` 调整）、`export default <推断对象> satisfies <TypeName>`。
- **推断（MVP）**：优先支持**字面量形状**可解析的场景；**空类型 / 空对象** → locale 为 `export default {} satisfies XxxMessage`。若 base 中类型已有**必填字段**，而推断结果为空对象，可能导致 **TS 不满足 `satisfies`**——属 MVP 已知边界，后续可做「按 key 补占位」等增强。
- **重命名实现注意**：文件系统常表现为 **unlink + add**，需在实现中用**短窗口内同目录事件关联**或 **debounce 后对比 base 目录快照**识别「改名」；可第二阶段再强化，MVP 可先保证 **增/删/改** 正确。

---

## 4. 实现风险（不重问，仅备案）

1. **推断与 TS 合法性**：必填字段与空 default 冲突时，locale 文件会报错，需后续策略（占位符、仅更新 key 集合等）。
2. **重命名**：依赖事件合并或快照 diff，与单纯「删 + 增」分时序有关。
3. **静默 MVP**：契约/缺失文件问题若不在终端与 overlay 暴露，**开发与 CI 可能长期静默失败**，后续需明确 **`build` 是否严格失败** 等开关。

---

## 5. 推荐重构顺序（从哪入手）

1. **`types` + `resolve-config`**  
   - 增加 **`framework`**（如 `'vite' | 'next'`，枚举可随项目扩展）。  
   - **未传 `i18nDir`** 时由 `framework`（及文档约定）给出默认（首版可统一默认 `src/i18n`，Next 特殊路径后续再分叉）。  
   - 业务逻辑**只使用** `resolveConfig` 后的绝对路径。

2. **新建 `core` 模块：base → 各 locale 同步**（与 Vite 解耦，便于单测）  
   - 输入：`ResolvedConfig` + 变更描述（增/删/改/重命名；MVP 可先做增删改）。  
   - 输出：对磁盘写/删/改名 locale 文件，内容规则见 §2、§3。

3. **接入变更入口**  
   - 在 **`I18nSourcesWatcher`**（或统一的 `scheduleRegenerate` 前）执行：**同步 → `generateAll`**。

4. **收敛 `notify` / `run-generate`（MVP 静默）**  
   - 例如 **`silent: true`** 或 **`diagnostics: 'none'`**：dev 不 overlay、不把 validation 当错误轰炸终端；具体字段名实现时再定。

5. **插件 `index.ts` 瘦身**  
   - `runGenerate`：**可选 scaffold → sync → generateAll →（可选）轻量日志**；watch / debounce 保持薄封装。

**一句话**：先从 **配置解析与 `framework` 默认值** 入手，再做 **core 层 base↔locale 同步**，接到 **chokidar + `generateAll`**，最后 **关掉 overlay/错误提示** 完成 MVP。

## 6. 功能清单（勾选跟踪）

- [ ] 引入 chokidar，插件内专职监听契约与各 locale（含 `i18n` 根、`ready` 与路径签名重同步）。
- [ ] `framework` 配置与默认 `i18nDir` 策略；用户任意相对根路径覆盖。
- [ ] dev 启动时无 i18n/locale 布局则脚手架（默认 `src` 下，与 §2 一致）。
- [ ] base 目录全方位监测驱动的 locale **增 / 删 / 改 / 重命名**（与 §3 一致，`satisfies` 风格）。
- [ ] 同步完成后仍由 **`generateAll`** 生成 `contracts.ts` 等产物。
- [ ] MVP：**关闭** overlay 与终端错误式提示（可留 `console.info` 类可选日志，实现时定）。
