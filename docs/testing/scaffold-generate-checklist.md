# i18next-kit：脚手架与生成逻辑 — 测试场景清单

在场景**完全跑通**后，将对应项由 `[ ]` 改为 `[x]`。

---

## 1. 脚手架开关与入口

- [ ] **1.1** `scaffold` 未传（默认）：宿主目录未就绪时，先创建最小模板再生成，不因缺目录在 `assert` 阶段失败。
- [ ] **1.2** `scaffold: false`：不写入任何脚手架文件；缺 `i18n` 根或契约目录时，`assertResolvedI18nLayout` 按设计抛错。
- [ ] **1.3** 仅调用 `generateAll`（不经 Vite 插件）、默认开启脚手架：行为与「插件内先 prepare 再生成」在结果上等价（有效 `i18n` 根与产物一致）。
- [ ] **1.4** Vite 插件：每次生成前用 `prepareI18nScaffold` 更新配置（含可能被改写的 `i18nDir`），再调用 `generateAll({ …, scaffold: false })`，避免重复脚手架且仍做布局断言。

---

## 2. 「已落地」判定（不自动脚手架）

- [ ] **2.1** `base` 下存在至少一个 `.ts` 契约文件（不含 `.d.ts`），且配置的每个 `locale` 目录均存在：不写入脚手架文件，生成直接使用当前 `i18n` 根。
- [ ] **2.2** `base` 目录存在但其中没有任何 `.ts` 契约：不视为已落地，进入脚手架/分类流程。
- [ ] **2.3** `base` 下仅有 `.d.ts`，无普通 `.ts`：不视为已落地（与实现「只认 `.ts` 且排除 `.d.ts`」一致）。
- [ ] **2.4** 契约侧看似齐全，但配置的某一 `locale` 目录缺失：不视为已落地，不应静默跳过脚手架分支。
- [ ] **2.5** 配置的 `i18n` 根路径在磁盘上不存在：不可能判定为已落地，应走创建/分类逻辑。

---

## 3. 分类：primary / new-locale / new-i18n

- [ ] **3.1** **primary**：配置的 `i18n` 根目录不存在，或目录存在但除忽略项外为空 → 在配置的 `i18nDir` 下写入最小模板。
- [ ] **3.2** **new-locale**：`i18n` 下仅有配置的 locale 子目录、且不存在契约目录（无 `base`）→ 在并列目录 `new-locale` 下写入模板，不覆盖原有 locale 树。
- [ ] **3.3** **new-i18n**：存在 `base`、或目录结构不满足「纯 locale 列表」（如杂项文件、空 `base` 等）→ 在并列目录 `new-i18n` 下写入模板。
- [ ] **3.4** 子目录名看似 locale，但其中混入非当前 `locales` 配置的文件夹：不满足「仅 locale」条件 → 行为与实现一致（通常为 `new-i18n` 或等价占用分支）。

---

## 4. Marker（`.i18next-kit-scaffold`）

- [ ] **4.1** 首次脚手架成功后，在**有效** `i18n` 根下写入 marker，且包含 `version`、`kind`、`configuredI18nRelative`。
- [ ] **4.2** 第二次运行：用户未改 `root`/`i18nDir` 且 marker 仍在 → 沿用上次选中的有效根（不在 primary 与 `new-*` 之间无故切换）。
- [ ] **4.3** 用户删除 marker 但模板文件仍在：再次运行时的分类/路径行为符合当前产品约定（可接受重新分类）。
- [ ] **4.4** 修改 `root` 或 `i18nDir` 导致与 marker 内 `configuredI18nRelative` 不一致：按实现重新解析有效根，不出现静默错误路径。

---

## 5. 幂等与覆盖

- [ ] **5.1** 对同一有效根多次执行默认 `generateAll`：已存在的脚手架文件（如 `common.ts`）不被覆盖；多次运行结果稳定。
- [ ] **5.2** 已落地项目多次生成：不产生脚手架写盘；源码未变时重复生成的 `writtenFiles` 可为空（与 `writeIfChanged` 行为一致）。

---

## 6. 配置组合

- [ ] **6.1** 使用默认 `i18nDir`（如 `src/i18n`）：空项目时在默认相对路径下生成模板与产物。
- [ ] **6.2** 自定义相对 `root` 的 `i18nDir`：分类、marker 相对路径与生成目标一致。
- [ ] **6.3** 自定义 `contractsDir`（相对或绝对）：脚手架写入的契约目录与 `resolveConfig` 解析结果一致；当有效根被改写到 `new-i18n` / `new-locale` 时，契约路径仍正确。
- [ ] **6.4** 未显式配置 `outDir`：产物目录等于解析后的有效 `i18nDir`（与当前默认策略一致）。

---

## 7. 校验与生成衔接

- [ ] **7.1** 脚手架刚写入最小 `common` 契约与各 locale 的 `common` 后：`validation` 在默认 locales 下可通过（无额外手工文件时）。
- [ ] **7.2** `scaffold: false` 且故意缺文件/缺 namespace：走既有 `validate` 行为，问题可被报告，不被脚手架掩盖。

---

## 8. 插件 / 开发态（可选 E2E）

- [ ] **8.1** 首次 `vite dev`：`src/i18n` 不存在或为空 → 自动生成模板与产物；终端/overlay 不出现误导性的「目录不存在」断言错误。
- [ ] **8.2** 首次 `vite dev`：仓库已是完整 kit 布局（契约 + 各 locale）→ 不创建 `new-i18n` / `new-locale`；监听路径为最终有效根。
- [ ] **8.3** 脚手架完成后修改某 locale 下的源文件：HMR/防抖触发重新生成，且仍使用更新后的 `i18nDir`（含 alternate 场景）。

---

## 9. 边界与负面

- [ ] **9.1** `locales` 为空数组或非法 `mode`：`resolveConfig` 抛出 `INVALID_CONFIG`，不进入脚手架写盘。
- [ ] **9.2** `mode: 'file'`（若当前仍禁止）：抛出配置错误，行为明确。
- [ ] **9.3** 磁盘无写权限或只读：脚手架或写入失败时有可观测错误，不表现为「成功但无文件」。

---

## 使用说明

- 建议顺序：先 **1 → 2 → 3 → 4**，再 **5 → 6 → 7**，**8** 在真实 Vite 项目中做，**9** 按需。
- 自动化单测已覆盖部分条目时，可在本清单旁备注用例文件与用例名（可选）。
- 若实现调整导致某条「预期」变化，请同步更新本文件中的描述，避免勾选含义漂移。
