# Phase 1：Core 纯逻辑

> 目标：把 `src/core/` 下的所有占位文件填满真实逻辑，实现「**给我一份 Config → 吐给你 4 份生成文件**」的纯函数闭环。
> 本 Phase **不涉及 Vite、不涉及 CLI**——所有东西都是无副作用（除写文件外）的 Node 纯模块，便于单元测试。
>
> **耗时预估**：1–2 天。核心难点在 P1-02 / P1-03 / P1-05 / P1-06，其余都是扁平。

## P1-01 `core/types.ts` — Config 与数据类型

### 目标
把整个 core 会用到的**类型、常量、错误类**全部在一个文件里定义好，作为后续所有模块的"词汇表"。

### 前置
Phase 0 完成

### 需要定义

#### 1. 用户面向的 Config
```ts
export interface I18nextKitConfig {
  /** 项目根目录，默认 process.cwd() */
  root?: string;
  /** i18n 源目录，相对 root，默认 'src/i18n' */
  i18nDir?: string;
  /** 契约目录，相对 i18nDir，默认 'base' */
  contractsDir?: string;
  /** 语言列表（目录名 / 文件名），必填 */
  locales: readonly string[];
  /** 实现层模式 */
  mode: 'folder' | 'file';
  /** 命名映射，默认 'kebabToCamel'，仅 file 模式有效 */
  nameTransform?: 'none' | 'kebabToCamel';
  /** 是否启用懒加载 runtime，默认 false */
  lazy?: boolean;
  /** 产物输出目录，默认同 i18nDir */
  outDir?: string;
  /** 各产物文件名覆盖（可选） */
  fileNames?: {
    resources?: string;   // 默认 'generated-resources.ts'
    contracts?: string;   // 默认 'contracts.ts'
    runtime?: string;     // 默认 'generated-runtime.ts'
    dts?: string;         // 默认 'i18next.d.ts'
  };
}
```

#### 2. 内部规整后的 ResolvedConfig
`resolveConfig(userConfig): ResolvedConfig` —— 把所有 optional 填上默认、把相对路径解析为绝对路径。后续模块只接受 `ResolvedConfig`。

```ts
export interface ResolvedConfig {
  root: string;           // absolute
  i18nDir: string;        // absolute
  contractsDir: string;   // absolute
  outDir: string;         // absolute
  locales: readonly string[];
  mode: 'folder' | 'file';
  nameTransform: 'none' | 'kebabToCamel';
  lazy: boolean;
  fileNames: Required<NonNullable<I18nextKitConfig['fileNames']>>;
}
```

#### 3. 扫描结果
```ts
export interface ScannedNamespace {
  /** namespace id，如 'common' / 'user-management' */
  name: string;
  /** base/common.ts 的绝对路径 */
  contractPath: string;
  /** base 里导出的类型名（猜测或用户显式约定，P1-02 详述） */
  typeName: string;
  /** 类型的 import 相对路径（相对 outDir） */
  typeImportPath: string;
}

export interface ScannedLocaleFile {
  locale: string;
  namespace: string;
  filePath: string;       // absolute
  importPath: string;     // 相对 outDir
}

export interface ScanResult {
  namespaces: ScannedNamespace[];
  locales: ScannedLocaleFile[];
}
```

#### 4. 错误类型
```ts
export class I18nextKitError extends Error {
  constructor(
    public readonly code:
      | 'CONTRACTS_DIR_NOT_FOUND'
      | 'LOCALE_DIR_NOT_FOUND'
      | 'MISSING_LOCALE_FILE'
      | 'MISSING_NAMESPACE'
      | 'EMPTY_CONTRACTS'
      | 'INVALID_CONFIG',
    message: string,
    public readonly detail?: Record<string, unknown>,

    super(message);
    this.name = 'I18nextKitError';
  }
}
```

## P1-02 `core/scan-contracts.ts`

### 目标
扫描 `contractsDir`，产出 `ScannedNamespace[]`。

### 前置
P1-01

### 逻辑

1. 读 `contractsDir` 下所有 `*.ts` 文件（不含子目录）
2. 对每个文件：
   - `name` = 文件名去掉 `.ts` 扩展
   - `contractPath` = 绝对路径
   - `typeName` 与 `typeImportPath` 的确定策略见下方
3. 按 `name` 字母序排序，返回数组
4. 若数组为空，抛 `I18nextKitError('EMPTY_CONTRACTS', ...)`
5. 若目录不存在，抛 `I18nextKitError('CONTRACTS_DIR_NOT_FOUND', ...)`

### `typeName` 确定策略

我们假设 `base/common.ts` 里会导出一个**叫做 `CommonMessage` 的类型**，这是当前 README 的约定。规则：

```
typeName = PascalCase(namespaceName) + 'Message'
```

举例：
- `common` → `CommonMessage`
- `user-management` → `UserManagementMessage`
- `table-columns` → `TableColumnsMessage`

如果未来要支持用户自定义后缀（如 `Contract` 结尾），留一个可扩展点，但**P1 不做**，只支持 `Message` 后缀。

### 不要做的事
- **不要**尝试 parse `base/*.ts` 的 AST 去验证 `typeName` 是否真的导出——这留给 TypeScript 编译器自己报错。core 只负责按约定猜名字并引用。
- 这样做的好处：core 零依赖，扫描快；代价：用户写错 typeName 会在 `contracts.ts` 编译时报错（可接受，错误信息清晰）。

### 实现提示
- 用 `fs.readdirSync(contractsDir, { withFileTypes: true })`，过滤 `isFile() && name.endsWith('.ts')`
- 忽略 `*.d.ts` 和以 `_` 开头的文件（预留内部文件可能）
- PascalCase 转换：`'user-management' → 'UserManagement'`，手写即可（不需要 lodash）
- `typeImportPath` 计算：`path.relative(outDir, contractPath)` → 处理 Windows 反斜杠 → 去 `.ts` 扩展 → 不以 `./` 开头时补上

### 验收（DoD）
- [ ] 给定 `base/common.ts` / `base/user-management.ts`，返回排序后的数组，两项 `typeName` 分别为 `CommonMessage` / `UserManagementMessage`
- [ ] 空目录抛 `EMPTY_CONTRACTS`
- [ ] 目录不存在抛 `CONTRACTS_DIR_NOT_FOUND`

## P1-03 `core/scan-locales-folder.ts`

### 目标
folder 模式下，扫描每个语言目录，产出 `ScannedLocaleFile[]`。

### 前置
P1-01

### 逻辑

对每个 `locale ∈ config.locales`：

1. 目录 = `path.join(i18nDir, locale)`

- `importPath`：相对 outDir，不带 `.ts` 后缀，Windows 路径要 normalize

## P1-04 `core/validate.ts`

### 目标
结合 contracts 与 locales 的扫描结果，做**完整性校验**，**不抛错**而是返回 `ValidationReport`，让上层决定怎么处理（严格 vs 宽松）。

### 前置
P1-02, P1-03

### 接口

```ts
export interface ValidationIssue {
  code: 'MISSING_LOCALE_FILE' | 'EXTRA_LOCALE_FILE';
  locale: string;
  namespace: string;
  expectedPath?: string;
  actualPath?: string;
}

export interface ValidationReport {
  ok: boolean;
  issues: ValidationIssue[];
}

export function validate(
  namespaces: ScannedNamespace[],
  locales: ScannedLocaleFile[],
  config: ResolvedConfig,
): ValidationReport;
```

### 检查规则

对 `namespaces × config.locales` 做笛卡尔积：

- 若 `(locale, namespace)` 在 `locales` 里不存在 → `MISSING_LOCALE_FILE`
- 若 `locales` 里存在但 `namespaces` 里没有对应 → `EXTRA_LOCALE_FILE`（警告级，不致命，但仍列出让用户清理）

**不做**的事：
- 不做 key-level 校验（key 级别的完整性由 TypeScript `satisfies` 编译期兜底，不在 core 里重复做）

### 验收（DoD）
- [ ] namespaces 有 `common`、`file`；locales 只有 `en-US/common.ts` → 返回 3 条 `MISSING_LOCALE_FILE`
- [ ] 多出 `en-US/legacy.ts` → 返回 1 条 `EXTRA_LOCALE_FILE`
- [ ] 完美对齐 → `{ ok: true, issues: [] }`

## P1-05 `core/emit/resources.ts`

### 目标
根据扫描结果生成 `generated-resources.ts` 的**字符串内容**（不写盘，返回 string；写盘在 orchestrate 里统一处理）。

### 前置
P1-02

### 产物样例（folder 模式、eager）

```ts
/* AUTO-GENERATED... */
import type { CommonMessage } from '../base/common';
import type { FileMessage } from '../base/file';
import type { UserManagementMessage } from '../base/user-management';

export const defaultNS = 'common' as const;

export const resourceNamespaces = {
  common: {} as CommonMessage,
  file: {} as FileMessage,
  'user-management': {} as UserManagementMessage,
} as const;

export type I18nNamespace = keyof typeof resourceNamespaces;
```

### 设计要点
- `defaultNS` 取 `namespaces[0].name`（字母序第一个）。**后续可以做配置项，P1 暂用这个简单策略**。
- `resourceNamespaces` 用 `{} as XxxMessage` 的假值——这是**纯类型占位**，运行时不会用它作为资源（真正资源由 runtime 加载），它只喂给 i18next 的 `CustomTypeOptions` 做类型推导。
- 带 `-` 的 key 必须加引号 `'user-management': ...`

### 接口
```ts
export function emitResources(
  namespaces: ScannedNamespace[],
  config: ResolvedConfig,
): string;
```

### 验收（DoD）
- [ ] 输出的 TS 字符串用 `ts.transpile` / `tsc` 能通过（手动 copy 到测试项目验证也行）
- [ ] `defaultNS` 为字母序第一
- [ ] `-` 命名空间 key 被加引号

## P1-06 `core/emit/contracts.ts`

### 目标
生成完全机械的 `contracts.ts`，让 TS 编译期强约束。

### 前置
P1-02, P1-03

### 产物样例

```ts
/* AUTO-GENERATED... */
import type { CommonMessage } from './base/common';
import type { FileMessage } from './base/file';

import enUSCommon from './en-US/common';
import enUSFile from './en-US/file';
import zhCNCommon from './zh-CN/common';
import zhCNFile from './zh-CN/file';

type LocaleContract = {
  common: CommonMessage;
  file: FileMessage;
};

export const contracts = {
  'en-US': { common: enUSCommon, file: enUSFile },
  'zh-CN': { common: zhCNCommon, file: zhCNFile },
} satisfies Record<'en-US' | 'zh-CN', LocaleContract>;
```

### 设计要点
- import 别名命名：`${camelCase(locale)}${PascalCase(namespace)}`
  - `en-US` → `enUS`，`common` → `Common` → 合成 `enUSCommon`
- 即使当前 mode 是 `file`，contracts 仍按 namespace 粒度导入——**file 模式下导入 `./${locale}` 的 namespace named export**，待 P3 实现时细化
- P1 只做 folder 模式，file 模式的 contracts 生成推后

### 接口
```ts
export function emitContracts(
  namespaces: ScannedNamespace[],
  locales: ScannedLocaleFile[],
  config: ResolvedConfig,
): string;
```

### 验收（DoD）
- [ ] 3 语言 × 2 namespace → 生成文件能过 `tsc`（配合 P1-11 的 fixture 验证）
- [ ] import 别名无冲突、符合驼峰规则
- [ ] satisfies 的 Record 键精确等于 `config.locales` union

## P1-07 `core/emit/runtime.ts`（eager 版本）

### 目标
生成 `generated-runtime.ts`，对外暴露 `initI18n(options?)` 函数。

**P1 只做 eager 版本**，懒加载版本放 Phase 3。

### 前置
P1-02, P1-03

### 产物样例（eager）

```ts
/* AUTO-GENERATED... */
import i18n, { type InitOptions, type Resource, type ResourceKey } from 'i18next';
import { initReactI18next } from 'react-i18next';
import { defaultNS, resourceNamespaces } from './generated-resources';

// 参与编译期契约校验，运行时无副作用
import './contracts';

const modules: Record<string, unknown> = import.meta.glob(
  './{en-US,zh-CN}/*.ts',  // 按 config.locales 动态拼接
  { eager: true },
);

const resources: Resource = { 'en-US': {}, 'zh-CN': {} };  // 按 locales 生成

for (const p in modules) {
  const match = p.match(/^\.\/(en-US|zh-CN)\/(.+)\.ts$/);  // 动态正则
  if (!match) continue;
  const [, locale, ns] = match;
  const mod = modules[p] as { default: ResourceKey };
  (resources[locale] as Record<string, ResourceKey>)[ns] = mod.default;
}

export function initI18n(options?: Omit<InitOptions, 'resources' | 'defaultNS' | 'ns'>) {
  return i18n.use(initReactI18next).init({
    resources,
    defaultNS,
    ns: Object.keys(resourceNamespaces),
    fallbackLng: 'en-US',
    interpolation: { escapeValue: false },
    ...options,
  });
}
```

### 设计要点
- glob 的 locale 列表、正则里的 locale alternation、resources 初始化对象 —— 都由 config.locales 动态生成
- 不要在 runtime 里 import `contracts` 的值，只 `import './contracts'` 做副作用参与编译
- `initI18n` 允许用户传除 `resources/defaultNS/ns` 外的所有 `InitOptions`

### 接口
```ts
export function emitRuntime(
  config: ResolvedConfig,
): string;
```

### 验收（DoD）
- [ ] 给定 locales=['en-US','zh-CN','zh-HK'] → glob pattern 正确，正则正确，resources 初始化正确
- [ ] 产物里没有把具体 namespace 写死（namespace 由 glob 自动收集）

## P1-09 `core/orchestrate.ts`

### 目标
对外暴露统一入口 `generateAll(config)`，串起 scan → validate → emit → 写盘。

### 前置
P1-04, P1-05, P1-06, P1-07, P1-08

### 接口

```ts
export interface GenerateResult {
  writtenFiles: string[];           // 绝对路径
  validation: ValidationReport;     // 让上层（plugin/cli）决定 warn 还是 error
  durationMs: number;
}

export async function generateAll(
  userConfig: I18nextKitConfig,
): Promise<GenerateResult>;
```

### 流程

1. `resolveConfig(userConfig)` — P1-01
2. `scanContracts(resolved)` — P1-02
3. `scanLocalesFolder(resolved)`（若 mode=file，P1 抛 `INVALID_CONFIG` 占位，Phase 3 补）— P1-03
4. `validate(ns, loc, resolved)` — P1-04（不抛，只收集）
5. 并行调用四个 `emit*` 得到四个字符串
6. 写盘：创建 `outDir`（若不存在），原子写每个文件（`tmp → rename`）
7. 记录 `writtenFiles`、返回

### 实现提示
- **原子写**：先写 `xxx.ts.tmp` 再 `rename`，避免 Vite 读到半截文件
- 写盘前对比**旧内容 hash**，内容一致则跳过（减少 HMR 噪声）
- 所有文件头统一 `AUTOGEN_HEADER`（带 contractsDir / 时间戳更方便 debug？——**不要带时间戳**，否则每次生成都是新 diff）

### 验收（DoD）
- [ ] 给定 fixture，调用后 `outDir` 里出现 4 个文件
- [ ] 第二次调用（输入不变）内容完全相同，`git diff` 为空
- [ ] `validation.ok === false` 时仍正常写盘（不抛，让上层决策）
- [ ] `INVALID_CONFIG`（mode='file' 时）抛 `I18nextKitError`
